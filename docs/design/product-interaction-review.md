## 3. 当前项目说明

### 3.1 产品定位

当前项目是一个面向个人用户的 AIGC 广告素材生成桌面工具，采用 Electron + React + SQLite 架构，本地保存任务和产物，通过云端模型 API 完成重型推理，不引入登录、多租户、计费等 SaaS 能力。

产品的核心价值不是“单条生成”，而是把用户输入的一条素材、一个需求、一组品牌资料，转化为一批可以继续筛选、确认、复用、投放的广告素材。

### 3.2 当前能力边界

结合 `spec.md` 与当前代码实现，产品目前已经形成以下一级模块：

- 广告文案脚本编写：输入需求，输出多版广告脚本
- 广告爆款素材裂变：输入抖音链接或本地视频，生成多版本裂变素材
- 原生爆款素材生成：面向六行业，输入 brief 生成完整素材
- 广告前贴生成：基于原广告生成开场钩子并拼接
- 数字人口播：基于数字人图、品牌介绍、产品图生成口播视频
- 飞书视频下载：从飞书文档中解析并下载视频
- 工作流可视化：查看 DAG 节点和默认 Prompt
- 素材库：查看本地生成产物
- 设置：管理模型、语音、存储、本地行为配置

### 3.3 当前产品结构

从现有前端结构看，产品已经具备一个完整工作台雏形：

- 首页：快捷进入各工作流，并查看最近任务
- 各模块页面：以表单方式创建任务
- 最近任务区：统一查看任务状态、总进度、基础操作
- 任务展开区：查看步骤级状态、产物、日志、重试、脚本确认
- 素材库：按记录查看本地产物路径
- 设置页：统一配置各种模型与服务参数

### 3.4 当前交互主链路

产品当前的主要使用路径可以概括为：

1. 进入某个功能模块
2. 填写表单并创建任务
3. 在任务列表中观察状态与进度
4. 在 `waiting_confirmation` 状态下确认脚本文案
5. 生成完成后，从任务产物或素材库中打开结果
6. 若失败，则重试任务或从某个节点继续执行

这个链路具备基础可用性，但仍偏“工程工具台”表达，距离“低心智负担的生产型产品”还有明显优化空间。

## 4. 当前交互现状判断

### 4.1 已有优点

- 模块结构清晰，主要能力已经分仓明确
- 所有任务统一进入任务队列，产品心智一致
- 任务支持步骤展开、日志、产物预览、重试、克隆、删除
- 脚本文案确认节点已经被产品化，而不是埋在后台流程中
- 工作流页可以帮助理解 DAG 结构，适合内部调试与迭代

### 4.2 当前主要问题

从产品角度看，当前问题不在“功能少”，而在“用户缺少确定感、过程缺少安心感、失败后缺少可操作感”。

主要体现为：

- 创建任务时，表单参数较多，但缺少任务预期说明
- 运行任务时，用户知道任务在跑，但不知道为什么还在跑
- 中间产物虽已存在，但在主流程中的曝光度不够
- 脚本确认虽然存在，但仍更像技术节点而不是审阅环节
- 失败提示和成功提示都偏弱，未形成明确下一步引导
- 素材库更像文件列表，还未形成“可复用资产中心”
- 设置页偏工程配置页，对非技术用户不够友好

## 5. 重点优化建议

以下建议按产品影响优先级排序。

### 5.1 优先级 P0：创建前降低不确定性

#### 建议 1：把“参数表单”改成“目标驱动表单”

当前各模块页面大多直接暴露输入参数，如时长、数量、比例、分辨率、brief 等。这种方式对熟悉系统的人有效，但对普通用户来说更像在填配置，而不是在表达需求。

建议改为两层结构：

- 第一层：你要什么结果
- 第二层：高级参数与生成控制

例如：

- 原生爆款：先问“你想做哪类素材”“主要卖点是什么”“希望投什么场景”
- 爆款裂变：先问“你要复刻节奏、改文案，还是做多版变体”
- 数字人口播：先问“你要强调品牌介绍、产品卖点，还是促销转化”

这样可以先建立结果导向，再承接技术参数。

#### 建议 2：增加任务创建前预检

当前创建任务后直接入队，但用户往往不知道：

- 当前配置是否齐全
- 这次任务大概会走哪些步骤
- 是否会进入脚本确认
- 预计耗时和可能失败点

建议在提交前加入简短预检区：

- 配置检查：缺少哪些 API Key 或必要设置
- 输入检查：素材格式、图片数量、时长范围是否合理
- 路径说明：本次任务会经过哪些节点
- 预计耗时：短、中、长
- 风险提示：例如参考图过少、brief 过短、平台信息缺失

这一步会明显降低“提交后才发现有问题”的挫败感。

#### 建议 3：提供示例与推荐值

`brief`、`brandIntro`、`requirement` 这类字段的结果高度依赖输入质量，但当前页面更多只是空文本框。

建议补充：

- 推荐输入模板
- 行业示例
- 优质输入示例
- 常见错误示例

这会直接提升首轮生成成功率。

### 5.2 优先级 P0：执行中增强安心感

#### 建议 4：把“总进度”升级为“当前节点说明”

当前任务列表能看到总进度百分比，但用户并不知道：

- 当前卡在哪一步
- 这一步在做什么
- 为什么需要这么久
- 下一步是什么

建议在任务主表层直接展示：

- 当前节点名称
- 当前节点作用说明
- 下一节点名称
- 已运行时长
- 预计剩余时间

例如：

- 正在执行：脚本生成
- 说明：系统正在根据行业模板和需求生成多版文案
- 下一步：脚本确认
- 已运行：1 分 42 秒

比单纯的进度条更能建立信任。

#### 建议 5：尽早暴露中间成果

当前中间产物主要存在于展开行和预览弹窗里。对长任务产品来说，这还不够。

建议将关键中间成果在主流程中前置展示：

- 文案任务：先显示模板匹配、研究信息、策略分析摘要
- 视频任务：先显示脚本、分镜、最终视频提示词
- 生成任务：先显示首个成功片段或首版结果

核心目标是让用户在等待最终成片前，持续获得“系统正在产出价值”的感知。

#### 建议 6：增加主动提醒机制

当前顶部主要提示“运行中数量”，但真正需要用户介入的场景是：

- 待确认
- 失败
- 已完成

建议增加统一提醒中心或显式状态条：

- 待确认任务数
- 失败任务数
- 刚完成任务数
- 最近可操作任务

同时支持桌面通知或轻量角标提醒，减少用户错过关键节点。

### 5.3 优先级 P0：把脚本确认做成真正的审阅节点

#### 建议 7：将脚本确认升级为“脚本审阅台”

脚本确认不是普通步骤，而是所有视频生成类工作流的人机协同核心节点。当前它已经有专门面板，但仍偏向“预览文本 + 点击确认”。

建议升级为具备以下能力的审阅界面：

- 左侧展示脚本结构和分镜段落
- 右侧展示合规提示、关键卖点、目标风格
- 支持局部编辑
- 支持局部重写
- 支持版本对比
- 支持标记“保留这段”“这一段太弱”

如果不想一版做太重，也建议至少先做：

- 一键复制脚本
- 重新生成当前脚本
- 修改后确认
- 看合规提醒后继续

这个节点优化后，会显著提升用户对成片质量的掌控感。

### 5.4 优先级 P1：失败反馈与恢复路径更可操作

#### 建议 8：错误提示改成“问题 + 原因 + 动作”

当前失败时主要还是状态和错误文本。对于产品用户来说，更重要的是“我该怎么办”。

建议错误反馈结构统一为：

- 发生了什么
- 可能原因
- 推荐动作

例如：

- 视频生成失败
- 可能原因：Seedance 鉴权失败或额度不足
- 推荐动作：检查设置页 Key，保存后从当前节点重试

再例如：

- 数字人图片校验失败
- 可能原因：图片中不是单人正脸，或清晰度不足
- 推荐动作：更换正面清晰单人图后重新提交

#### 建议 9：把重试动作语义化

当前产品支持整任务重试和节点重试，这是很强的能力，但用户未必理解两者区别。

建议把通用图标按钮升级为带语义的动作：

- 重新执行整个任务
- 从脚本节点继续
- 仅重跑合规检查
- 修正配置后继续

重试动作越具体，恢复成本越低。

### 5.5 优先级 P1：成功后的结果消费闭环

#### 建议 10：任务完成后主动给出下一步

当前任务成功后，用户主要是看到状态变为完成，然后自己去找产物。

建议成功后直接给出结果型操作：

- 预览成片
- 在 Finder 中定位
- 打开素材库
- 克隆为相似任务
- 再生成 3 版
- 复制脚本

对于广告生产工具来说，生成成功不是终点，二次筛选和继续裂变才是更真实的后续行为。

### 5.6 优先级 P1：把素材库从文件列表升级为资产中心

#### 建议 11：增强素材库的信息维度

当前素材库主要展示类型、标签、路径，更偏工程结果索引。

建议补充：

- 缩略图或首帧预览
- 来源任务
- 所属模块
- 行业标签
- 创建时间
- 可复用入口

这样用户能更快判断哪些内容值得继续使用。

#### 建议 12：沉淀中间产物资产

除了最终视频，以下内容也应该逐步资产化：

- 脚本
- 分镜
- 研究结果
- Prompt
- 高质量开头钩子

一旦这些内容可复用，产品价值就会从“生成工具”提升为“创意资产工作台”。

### 5.7 优先级 P2：设置页降低工程感

#### 建议 13：拆分为基础配置和高级配置

当前设置页把模型、语音、对象存储、本地行为全部平铺，适合研发或深度用户，但对普通用户学习成本较高。

建议拆分为：

- 基础配置：最小必填项
- 服务测试：测试 LLM、视频、TTS、ASR 是否可用
- 高级配置：Base URL、Model ID、并发、默认值

#### 建议 14：增加连通性测试

保存设置不等于设置可用。建议每个服务增加“测试连接”或“验证配置”能力，避免用户在任务执行时才发现设置有误。

## 6. 页面级优化方向

### 6.1 首页

建议从“功能入口页”升级为“工作台首页”：

- 优先展示待确认、失败、刚完成任务
- 按场景提供快捷入口，而不只是按模块名导航
- 增加最近使用模板、最近使用参数、最近复用素材

### 6.2 任务列表

建议强化任务列表的决策效率：

- 默认显示当前节点和当前动作
- 待确认任务置顶
- 失败任务突出原因和恢复路径
- 完成任务支持一键预览与再生成

### 6.3 工作流页

当前工作流页更像内部调试工具，建议保留其工程属性，同时补充对产品用户更友好的说明：

- 节点含义
- 每一步输入与输出
- 哪一步需要用户参与
- 哪一步最耗时

### 6.4 各任务创建页

建议统一形成一套创建页设计原则：

- 先表达目标，再填参数
- 必填项前置，高级项折叠
- 关键字段给示例
- 创建前做预检
- 提交后告知后续链路

## 7. 建议的迭代顺序

### 第一阶段：低成本高收益

- 增加任务预检
- 增强任务运行中节点反馈
- 增强失败文案与推荐动作
- 增加待确认和失败提醒
- 成功后增加下一步操作入口

### 第二阶段：体验跃迁

- 升级脚本确认为审阅台
- 改造创建页为目标驱动式表单
- 强化素材库的筛选、预览和复用能力

### 第三阶段：资产化与智能化

- 把脚本、分镜、Prompt 逐步沉淀为可复用资产
- 基于历史任务提供推荐参数和快捷复用
- 提供按行业、场景、目标的创建模板

## 8. 总结

当前项目已经具备较完整的产品骨架和可运行任务链路，核心问题不在于缺少模块，而在于用户体验仍偏“工程工具操作台”。

下一阶段最值得投入的方向不是继续增加更多工作流，而是优先把以下三件事做好：

- 创建前让用户更确定
- 执行中让用户更安心
- 关键节点和失败场景让用户更可操作

如果这三点得到明显改善，产品会从“能跑起来的 AI 生成工具”进一步升级为“可持续使用的广告生产工作台”。

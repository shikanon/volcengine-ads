# 广告视频打分工作流开发计划

## 目标

基于 `docs/广告视频打分Prompt_内部标准增强版.md` 新增一个可运行的“广告视频打分”一级工作流：用户上传本地视频、手动选择广告类型（品牌/买量/创意），主进程直接调用 `ModelClient.visionVideo(videoPath, prompt)` 输出结构化评分结果，前端在任务详情里展示动态雷达图、条形图、证据、分析与优化建议。

## 涉及 spec 章节

- `spec.md §3` 工作流 DAG
- `spec.md §5` 输入契约
- `spec.md §8` Pipeline 产物与步骤约束

## Summary

- 新增独立任务类型 `video_scoring`，不复用现有 `copywriting` / `native` / `explosion`。
- 本期输入仅支持本地视频文件，表单要求用户手动选择广告类型，不实现自动分类、不实现同类型对比。
- 评分步骤直接按类型路由到对应 Prompt；即使合规不通过，也输出可展示的局部结果，前端据此展示合规提示和建议。
- 前端图表不新增第三方图表库，使用 React + SVG/CSS 实现雷达图和条形图，避免无必要依赖变更。
- 数据库存储、IPC 通道、任务仓储结构均可复用现状，无需 migration、无需新增专门 IPC。

## Current State Analysis

- 当前一级任务类型定义在 `src/shared/types.ts`，包含 `explosion`、`pretrailer`、`avatar`、`native`、`copywriting`、`lark_download`，尚无“视频评分/打分”类型。
- 工作流定义集中在 `src/shared/workflows.ts`，负责节点 DAG 和 Prompt 模板；`src/renderer/pages/Workflows.tsx` 直接消费这些定义。
- 主进程 pipeline 注册在 `src/main/pipelines/index.ts`，每个工作流对应单独目录；`src/main/pipelines/copywriting/index.ts` 是当前最接近“结构化 JSON -> markdown/report -> 任务预览”的可复用实现模板。
- 任务创建统一走 `task:create`，校验在 `src/main/validation.ts`，任务入库使用通用 `tasks.input_json`，所以新增任务类型不要求 DB schema 变更。
- 任务详情预览集中在 `src/renderer/components/TaskTable.tsx`；目前只有 `copywriting` 为 JSON 产物提供定制化可视化预览，没有现成图表能力。
- 依赖中没有 `echarts` / `recharts` 等图表库，`package.json` 当前也没有必要的可复用图表依赖，因此本次采用自绘 SVG 更符合现状。
- `spec.md` 当前只覆盖现有几类工作流，若新增 `video_scoring`，必须同步补 DAG、输入契约和产物约定，否则会违反仓库规则。

## Assumptions & Decisions

- 已确认本期范围是“完整新工作流”，不是单纯补 Prompt 或只做主进程能力。
- 已确认输入仅支持本地视频，不支持抖音链接，也不接入历史任务对比。
- 已确认广告类型由用户在表单中手动选择，不引入分类 Prompt，不做人工覆盖重跑流程。
- 已确认当合规不通过时任务仍然成功产出结构化结果；结果允许没有维度分数，但必须返回 `compliancePass`、问题说明、分析与建议。
- 图表采用仓内实现，不新增第三方图表依赖；若评分维度为空，则前端隐藏图表区域，改为展示合规拦截说明。
- 不新增 IPC channel，继续复用 `task:create` / `task:list` / `asset:*`；不新增数据库表和 migration。

## Proposed Changes

### 1. 规格与共享契约

- `spec.md`
  - 新增 `video_scoring` 一级任务类型说明。
  - 在 DAG 章节补充该工作流的步骤、产物和说明。
  - 新增输入契约，明确只支持本地视频 + 手动选择类型。
  - 新增输出契约，明确结构化 JSON 至少包含：
    - `category`
    - `compliancePass`
    - `complianceIssues`
    - `dimensionScores`
    - `evidence`
    - `analysis`
    - `suggestions`
  - 明确 `dimensionScores` 按类型动态变化，不做跨类型总分，不做跨类型对比。
- `src/shared/types.ts`
  - 扩展 `TaskType`，加入 `video_scoring`。
  - 新增 `AdVideoScoringCategory` 联合类型，建议值为 `brand | performance | creative`。
  - 新增 `VIDEO_SCORING_CATEGORY_DEFINITIONS` 或等价常量，供前端表单与渲染复用中文文案。
  - 新增 `VideoScoringInput`：
    - `sourceVideoPath: string`
    - `category: AdVideoScoringCategory`
  - 将 `TaskRecord.input`、`PipelineInput`、`CreateTaskRequest.input` 联合补齐到 `VideoScoringInput`。
  - 如实现中有价值，可补共享结果类型 `VideoScoringResult`，避免主/渲染端重复定义。

### 2. 工作流定义与 Prompt 模板

- `src/shared/workflows.ts`
  - 新增工作流 `video_scoring`，建议节点如下：
    1. `ingest`：素材导入，产物 `source.mp4`
    2. `score`：视频评分，产物 `score.json`
    3. `report_writer`：结果整理，产物 `report.md`
  - 新增三个 Prompt 定义：
    - `video_scoring.brand_score`
    - `video_scoring.performance_score`
    - `video_scoring.creative_score`
  - Prompt 内容基于 `docs/广告视频打分Prompt_内部标准增强版.md`，但统一输出契约：
    - `category`
    - `compliancePass`
    - `complianceIssues`
    - `dimensionScores`
    - `evidence`
    - `analysis`
    - `suggestions`
  - 其中 `score` 节点的 `promptIds` 同时挂三个 Prompt，便于工作流页调试查看。
  - 不加入分类 Prompt，因为产品决策已锁定为手动选择类型。

### 3. 主进程任务校验与 pipeline

- `src/main/validation.ts`
  - 新增 `validateVideoScoring(input)`。
  - 校验点：
    - `sourceVideoPath` 必填且文件存在
    - `category` 必须为 `brand | performance | creative`
  - 在 `validateCreateTaskRequest()` 中接入 `video_scoring` 分支。
- `src/main/pipelines/types.ts`
  - 将 `VideoScoringInput` 纳入 `PipelineInput` 联合。
- `src/main/pipelines/index.ts`
  - 注册 `videoScoringPipeline`。
- `src/main/pipelines/video-scoring/index.ts` 新增
  - `runIngest(ctx)`：
    - 复用 `normalizeVideo()` 把输入视频写为 `source.mp4`
    - 不抽帧，不额外做 ASR
  - `runScore(ctx)`：
    - 根据 `ctx.input.category` 选择对应 Prompt ID
    - 调用 `ctx.modelClient.visionVideo(artifactPath(ctx.artifactDir, 'source.mp4'), prompt)`
    - 解析 JSON 并做结果兜底/规整：
      - `dimensionScores` 只保留数值项
      - `evidence` 只保留字符串项
      - `suggestions` 归一为字符串数组
      - `compliancePass` 缺失时默认 `true`
      - 合规不通过但无分数时允许 `dimensionScores = {}`
    - 写入 `score.json`
  - `runReportWriter(ctx)`：
    - 读取 `score.json`
    - 生成便于素材库查看的 `report.md`
    - 注册一个 `report` 类型 asset，tag 至少包含 `video_scoring` 和具体 `category`
  - 结果 markdown 建议包含：
    - 类型与合规状态
    - 各维度分数表
    - 证据时间点
    - 模型分析
    - 优化建议
  - 不修改 `src/main/model-client/**`，直接复用现有 `visionVideo()` 能力。

## 具体执行顺序

1. 先改 `spec.md`，补齐 `video_scoring` 的 DAG、输入/输出契约和步骤命名。
2. 扩展 `src/shared/types.ts` 与 `src/shared/workflows.ts`，完成共享类型、工作流、Prompt 模板、页面可读元数据。
3. 实现 `src/main/validation.ts` 与 `src/main/pipelines/video-scoring/index.ts`，并在 pipeline 注册表接入。
4. 新增 `src/renderer/pages/VideoScoring.tsx`，接入 `App.tsx` 与 `Home.tsx`。
5. 在 `TaskTable.tsx` 和 `styles.css` 中实现评分结果可视化。
6. 补 unit / e2e 测试，最后执行 typecheck、lint、test、build。

## 风险与回滚

- 最大风险是模型返回 JSON 不稳定，导致图表渲染字段缺失；通过主进程 `runScore()` 的归一化和前端空态兜底控制风险。
- 第二风险是动态图表在无第三方库下样式复杂度上升；通过限制本期能力为“单任务单结果展示 + SVG 基础图形”控制实现成本。
- 若上线后发现图表表现不达预期，可保留 `score.json` + `report.md` 结构不变，仅回退前端到分数卡片/列表视图，不影响主进程产物契约。
